5 · 数据落点与同步模型（实现约定）

回答「记录存哪」：分三层。同步算法直接沿用 docs/web-server.md 已定的三方 hash 比对模型， 服务端 API（/index 带 hash、PUT/DELETE 带 If-Match）现在就够用，无需新增文件接口。

① 服务器 · 同步空间注册表
GET/PUT /api/sync/spaces（新增）

[{ id, name, remotePath, createdAt }]
哪些空间开启过同步。换电脑连同一服务器就能看到并拉取。

② 工作区内 · 绑定 + 索引
<folder>/.quill/sync.json

{ spaceId, serverUrl, files: { path: hashAtLastSync } }
本地文件夹 ↔ 空间的绑定关系和上次同步指纹。文件夹整体拷到新机器也能识别。

③ 桌面端 · 全局偏好
electron-store（现有 prefs）

{ autoSync: true, intervalMin: 5 }
自动同步开关与间隔。设备级设置，不随空间走。

状态判定（沿用已有设计）

状态判定归入
syncedlocal == remote == lastSync—
local-modified / local-only / local-deleted仅本地偏离 lastSync待推送 ↑
remote-modified / cloud-only / remote-deleted仅云端偏离 lastSync待拉取 ↓
conflict两端都偏离 lastSync手动处理

推送一律带 If-Match；412 即转冲突，绝不覆盖。 被覆盖侧先落 *.conflict-<时间>.md 备份。

关键交互流程

  1. 01 打开本地文件夹 → 发现 .quill/sync.json 且服务器可达 → 自动恢复同步状态，后台静默检查一次。
  2. 02 首次开启 → 在云端 /<目录> 创建空间 + 注册表登记 → 整体首次推送。
  3. 03 ⌘S 保存只写本地（现状不变）；保存后将该文件标记 dirty，指示器变 accent。自动同步到点或手动点推送才上云。
  4. 04 换电脑 → 设置或面板里看到服务器上的空间 → 「拉取到本地…」选目标文件夹 → 写入 .quill/sync.json，之后照常推拉。
  5. 05 单文件模式 / 远程模式：不出现同步指示器（远程模式本身就是直接读写云端）。
